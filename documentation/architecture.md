# Архитектура

## Общая схема

Приложение представляет собой клиентскую одностраничную игру. Основная логика сосредоточена в `App.tsx`, а рендеринг и интеграция с Yandex Maps вынесены в отдельные модули.

Бэкенда у проекта нет. Генерация раундов, проверки, подсчет очков и итоговая статистика выполняются прямо в браузере.

## Основные слои

### 1. Прикладной слой

`App.tsx` управляет:

- инициализацией API
- выбором режима игры
- переходами между игровыми состояниями
- генерацией раундов
- префетчем следующего раунда
- накоплением очков
- итоговой статистикой
- шарингом результатов

По сути это главный state machine всего приложения.

### 2. Слой представления

Интерфейс разбит на два основных компонента:

- `components/StreetView.tsx`
- `components/MiniMap.tsx`

`StreetView` отвечает за отрисовку панорамы. `MiniMap` отвечает за интерактивную карту, постановку ответа и визуализацию разницы между ответом игрока и реальной точкой.

### 3. Интеграционный слой

`lib/yandexMaps.ts` инкапсулирует детали работы с Yandex Maps:

- загрузку внешнего скрипта
- получение API-ключа
- поиск панорам
- nearby geocoding search

Это снижает связность UI-компонентов с внешним API.

### 4. Доменный и конфигурационный слой

`lib/gameConfig.ts` и `lib/mapUtils.ts` описывают:

- общие типы
- границы игровых режимов
- seed-локации для режима `SAKHA`
- формулу подсчета очков
- географические утилиты
- правила валидации населенных мест

## Поток выполнения

### Запуск приложения

1. `index.tsx` рендерит `<App />`
2. `App.tsx` вызывает `loadYandexMaps()`
3. `lib/yandexMaps.ts` получает API-ключ
4. В страницу добавляется скрипт Yandex Maps
5. После `ymaps.ready()` приложение создает сервисные обертки и разрешает игровой поток

### Генерация раунда

Генератор раунда работает по схеме повторных попыток:

1. Определяются границы для выбранного режима
2. Генерируется случайная кандидатная координата
3. Для нее ищется ближайшая панорама
4. Если панорамы нет, попытка отклоняется
5. Если найденная панорама вышла за пределы допустимых границ, попытка отклоняется
6. Выполняется nearby geocoding search
7. Если место не похоже на населенный пункт, попытка отклоняется
8. При успехе раунд принимается и сохраняется как пара `location + panorama`

Эта логика реализована в `generateValidLocation()` внутри `App.tsx` и использует вспомогательные функции из `lib/mapUtils.ts` и `lib/yandexMaps.ts`.

### Ответ игрока и скоринг

1. `MiniMap.tsx` слушает клики по карте, пока `gameState === "GUESSING"`
2. Координата клика становится `guessLocation`
3. Игрок подтверждает ответ
4. `App.tsx` считает расстояние через `haversineKm()`
5. `App.tsx` переводит расстояние в очки через `scoreFromDistance()`
6. Экран результата показывает очки, дистанцию и линию между guess и target

### Финальная статистика

После `TOTAL_ROUNDS = 5` приложение считает:

- суммарный XP
- средний счет
- общую дистанцию
- лучший раунд
- историю по каждому раунду

Эти данные выводятся в итоговом оверлее и могут быть отправлены через native share или скопированы в буфер обмена.

## Модель состояния

### Базовое состояние

Проект использует React state без внешнего state manager.

Ключевые поля состояния в `App.tsx`:

- `language`
- `gameMode`
- `gameState`
- `targetLocation`
- `targetPanorama`
- `prefetchedRound`
- `guessLocation`
- `distance`
- `score`
- `totalXP`
- `currentRound`
- `roundHistory`
- `loadingMessage`
- `error`

### Игровые состояния

Описаны в `lib/gameConfig.ts`:

- `MODE_SELECT`
- `GUESSING`
- `LOADING_RESULT`
- `RESULT`
- `FINAL_RESULT`

Типовой сценарий переходов:

```text
MODE_SELECT
  -> LOADING_RESULT
  -> GUESSING
  -> LOADING_RESULT
  -> RESULT
  -> LOADING_RESULT -> GUESSING
  -> FINAL_RESULT
```

## Ответственность по файлам

### `App.tsx`

Отвечает за:

- глобальное состояние игры
- старт новых раундов
- валидацию сгенерированных локаций
- вычисление результата
- запуск префетча
- рендер верхнеуровневого интерфейса и оверлеев

Важные детали реализации:

- `generationLock` не дает стартовать нескольким генерациям одновременно
- префетч хранится и в React state, и в refs для более отзывчивого поведения
- клавиша `Space` переводит экран `RESULT` к следующему раунду

### `components/StreetView.tsx`

Отвечает за:

- создание и уничтожение `ymaps.panorama.Player`
- отображение активной или предзагруженной панорамы
- показ fallback-сообщения, если панорама не создается
- overlay в верхнем левом углу для блокировки встроенной ссылки Yandex

Скрытая префетч-панорама рендерится вне видимой области, чтобы следующий раунд открывался быстрее.

### `components/MiniMap.tsx`

Отвечает за:

- создание мини-карты Yandex
- обработку кликов игрока
- постановку красной и зеленой меток
- рисование линии между guess и target
- расширение карты по hover и на экране результата
- подгонку viewport после анимированного изменения размеров

### `lib/yandexMaps.ts`

Отвечает за:

- одноразовую загрузку Yandex Maps
- чтение API-ключа из env или `api-key.txt`
- обертку над поиском панорамы
- обертку над nearby geocoding search
- кеширование nearby search по округленным координатам и радиусу

### `lib/mapUtils.ts`

Отвечает за:

- географические вычисления
- генерацию случайных координат
- распределение точек вокруг seed-локаций для `SAKHA`
- преобразование типов мест
- фильтрацию населенных мест
- формулу скоринга

### `lib/gameConfig.ts`

Отвечает за:

- общие типы
- границы режимов
- seed-локации
- допустимые типы мест

## Логика координат и регионов

### Границы

Границы режимов заданы в `BOUNDS_BY_MODE`:

- `YAKUTSK`: компактная область города
- `SAKHA`: большая область по республике

### Генерация для режима `SAKHA`

Режим `SAKHA` не использует равномерное случайное распределение по всей республике. Вместо этого выбирается одна из `SAKHA_LOCATION_SEEDS`, а затем точка генерируется рядом с ней.

Это сделано осознанно, потому что:

- территория очень большая
- покрытие панорамами редкое
- шансы найти пригодные игровые точки выше возле заранее известных населенных зон

### Проверка населенности

Nearby geocoding считается успешным только если хотя бы один результат имеет допустимый тип из `ALLOWED_PLACE_TYPES`, например:

- `locality`
- `political`
- `sublocality`
- `neighborhood`
- `administrative_area_level_3`

Такой фильтр уменьшает шанс отправить игрока в пустую местность.

## Модель скоринга

Текущая формула подсчета очков из `lib/mapUtils.ts`:

```ts
Math.max(0, Math.round(5000 * Math.exp(-distanceKm / 400)))
```

Что это дает:

- почти идеальные ответы дают значения около `5000`
- очки убывают плавно, а не ступенчато
- отрицательные значения невозможны

В `App.tsx` до сих пор есть дублирующие helper-функции из более ранней версии логики. Фактические расчеты сейчас идут через импортированные shared-функции, поэтому именно код из `lib/mapUtils.ts` является актуальным источником истины.

## Стратегия работы с API-ключом

Приложение поддерживает два источника ключа:

1. `VITE_YANDEX_MAPS_API_KEY`
2. `api-key.txt`

Сначала проверяется env, затем выполняется попытка загрузить `api-key.txt` с текущего origin.

Это удобно для:

- локальной разработки через `.env`
- статического деплоя с отдельным файлом ключа
- runtime-инъекции в Docker-контейнере

## Технический долг и риски

### Слишком много логики в `App.tsx`

`App.tsx` сейчас совмещает:

- текстовые ресурсы UI
- state machine
- генерацию локаций
- асинхронную оркестрацию
- рендер интерфейса

Со временем такой файл будет все сложнее поддерживать. Логично вынести игровую логику в хуки или отдельные сервисы.

### Дублирование логики

В `App.tsx` остались helper-функции и сервисные классы, которые частично пересекаются с `lib/mapUtils.ts` и `lib/yandexMaps.ts`.

Это повышает риск:

- рассинхронизации формул
- появления мертвого кода
- путаницы при поддержке

### Проблемы кодировки

Часть русских и саха строк в исходниках выглядит как текст с нарушенной кодировкой. Перед дальнейшей локализацией стоит проверить фактический вывод в браузере.

### Нет автоматических проверок качества

В репозитории сейчас нет настроенных:

- unit-тестов
- интеграционных тестов
- lint-скрипта
- format-скрипта

## Рекомендации по развитию

- вынести генерацию раундов в отдельный хук или сервис
- полностью отделить локализацию в специализированный i18n-модуль
- удалить устаревшие дубли в `App.tsx`
- добавить тесты для скоринга, границ и вспомогательной географической логики
- настроить линтер и форматтер
